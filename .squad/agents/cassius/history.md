## Learnings

- **httpx Response Mocking Pattern (2026-05-31):** When mocking httpx async client calls, mock the CLIENT's async methods (`.post()`, `.get()`, etc.) as `AsyncMock`, but mock the RESPONSE object's methods (`.json()`, `.raise_for_status()`, `.text`) as regular synchronous `Mock` instances. The response object in httpx exposes synchronous methods even when obtained from an async context manager.

- **Image Processing Utility Pattern (2026-05-31):** Created `src/api/capture/` package for circular image clipping (Issue #216 scope item 3). Self-contained stdlib-only utility (`image`, `image/color`, `image/png`, `image/jpeg`, `math`). Implements `ClipToCircle(src image.Image, guide CircleGuide) *image.RGBA` with anti-aliased edge (linear alpha blend over final 1px ring), premultiplied color channels for correct compositing. `ClipBytesToCirclePNG(data []byte, guide CircleGuide) ([]byte, error)` convenience encoder handles JPEG/PNG input via blank imports. `DefaultGuide` matches frontend viewfinder geometry (50%/52% center, 74% width, 360px cap). **Not yet wired into intake pipeline** â€” standalone primitive awaiting Brian's decision on which captured images get clipped (coin card must remain rectangular for OCR). Test pattern: geometry resolution, output sizing, corner transparency, center opacity, anti-alias band partial alpha, bounds intersection safety, round-trip encoding (JPEG+PNG), invalid data error handling.

- **CircleClip Handler Integration (2026-05-31):** Wired circular clipping into image upload handlers (Issue #216). Added `circleClip` boolean field to both `Upload` (multipart form) and `UploadBase64` (JSON) endpoints. Semantics: when `circleClip=true` AND `imageType` is `obverse` or `reverse`, decode uploaded bytes, run `capture.ClipBytesToCirclePNG(data, capture.DefaultGuide)`, store the resulting transparent PNG. Card/detail/other types never clipped (card must remain rectangular for OCR). Decode-error fallback: if clipping fails (e.g., undecodable bytes), log a warning and store the original image â€” preserves upload robustness, avoids 500 errors. Architecture test updated to allow `handlers/` â†’ `capture/` import (capture is stdlib-only utility). Comprehensive test suite (7 focused tests) covers: obverse clipped to PNG with transparent corners, detail NOT clipped, obverse without circleClip unchanged, undecodable data fallback for both multipart and base64 paths. All tests pass; full suite clean.

- **Ownership-Before-Decode Security Rule (2026-05-31):** Issue #216 hardening (post-code-review feedback): In upload handlers that decode/process attacker-controlled bytes (image decode, clip, etc.), ALWAYS validate coin ownership BEFORE any decode/process operations. Original clip integration decoded bytes first, then passed to service layer which checked `FindCoinByOwner()`. This meant an authenticated user could trigger CPU-intensive decode on any coin ID, even for coins they don't own (auth + 30MB cap mitigated, but contract Â§5 requires clip AFTER validation). Fix: added early `h.repo.FindCoinByOwner(coinID, userID)` check in both Upload/UploadBase64 handlers before file read or base64 decode when `circleClip=true`. Returns same 404 error as service layer, but without decoding untrusted bytes. Service layer still validates (defense in depth). Pattern: for any handler operation that consumes significant CPU/memory on untrusted input, gate it with ownership check first. Test pattern: upload with `circleClip=true` for non-owned coin â†’ 404 rejection, no file written, clip never executed. Bonus: added early `file.Size > 20MB` check before `io.ReadAll()` in multipart path for fail-fast on oversized uploads.
