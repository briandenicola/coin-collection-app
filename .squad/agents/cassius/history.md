- **2026-05-22:** **BUGFIX — Auction ending scheduler missing lots with null sale_date.** Brian reported a Heritage Auctions lot (Lot #8325, sale date May 22, 2026, status BIDDING) that was not flagged by the scheduler. Root cause: `AuctionLotRepository.GetEndingToday()` only checked `sale_date` field, excluding lots with `sale_date = NULL` even if `auction_end_time` was set to today. This affected non-NumisBids auctions (e.g., Heritage) where `ParseSaleDate()` fails or lots are created manually without a sale date. **Fix:** Updated query to check BOTH `sale_date` and `auction_end_time` fields with explicit NULL guards: `WHERE status = 'bidding' AND ((sale_date IS NOT NULL AND sale_date >= X AND sale_date < Y) OR (auction_end_time IS NOT NULL AND auction_end_time >= X AND auction_end_time < Y))`. Added test case "bidding lot with auction_end_time today (no sale_date)" to `auction_lot_repository_test.go` to prevent regression. All tests pass (`go test -v ./...` clean). **Impact:** Scheduler now catches Heritage Auctions and other non-NumisBids lots that populate `auction_end_time` instead of `sale_date`. See decision doc `.squad/decisions/inbox/cassius-auction-ending-null-date-fix.md` for full analysis.

- **2026-05-22:** **SECOND-PASS DIAGNOSIS — Auction ending scheduler STILL returns 0 lots after NULL-date fix.** Brian deployed the previous fix and re-ran the manual trigger. Still 0 lots. Identical 10ms duration is suspicious. **AUDIT FINDINGS:** (1) AuctionLot model has THREE date fields: `SaleDate`, `AuctionEndTime`, and `EventID` (foreign key to AuctionEvent which has `StartDate` and `EndDate`). **The displayed sale date in the UI might come from the linked AuctionEvent, NOT the lot's own date fields.** (2) Status filter uses lowercase `models.AuctionStatusBidding` constant = "bidding" — verified enum is lowercase. (3) No user scope filter found — scheduler iterates all users with Pushover enabled. **CRITICAL DISCOVERY:** Heritage lots likely have `EventID` set but both `SaleDate` and `AuctionEndTime` are NULL. The end date lives on the parent `AuctionEvent.EndDate` field. **SOLUTION:** Built a debug endpoint (`GET /api/admin/auction-ending/debug`) that returns: (a) total lots in DB, (b) lots by status, (c) lots matching current scheduler query, (d) ALL BIDDING lots with ALL date fields populated INCLUDING event dates (via LEFT JOIN). This diagnostic endpoint exposes the ground truth — which field actually holds Brian's lot's end date. **SQL query for Brian to run immediately:** `SELECT id, user_id, status, lot_number, sale_date, auction_end_time, event_id, created_at, updated_at FROM auction_lots WHERE lot_number = 8325 OR status = 'bidding' ORDER BY updated_at DESC LIMIT 10;` **LESSON:** Never ship a query fix without first inspecting real production data. Last fix was a guess based on schema alone. This time: (1) added debug endpoint, (2) provided SQL query, (3) NO query changes until Brian shares actual data. Added `AuctionLotRepository.GetAllBiddingLotsWithEventDates()` method (delegates raw SQL to repo for architecture compliance). All tests pass (`go vet` clean, `go test -v ./...` clean).

- **2026-05-22 (FINAL FIX):** **Root cause confirmed — UTC calendar-day semantic drops lots that end at midnight UTC.** Brian ran SQL and confirmed lot #8325 has `sale_date = 2026-05-23 00:00 UTC` (which is 2026-05-22 19:00 Central = 7 PM his time). The query used `[startOfDayUTC, endOfDayUTC)` range (`< endOfDay` is exclusive), so lots ending exactly at `2026-05-23 00:00` were excluded. Users in negative-offset time zones (US) see "today" but UTC sees "tomorrow." **TWO BUGS FIXED:** (1) Changed semantic from "ends on UTC calendar day" to "ends within next 24 hours" — query now uses `(now, now+24h]` window (strict lower bound to exclude already-ended auctions, inclusive upper bound to catch edge cases). Renamed method from `GetEndingToday()` to `GetEndingSoon()` to reflect the new semantic. (2) Hardened status comparison with `LOWER(status) = 'bidding'` to avoid future case-drift. **CHANGES:** Updated `auction_lot_repository.go` (new query), `auction_ending_scheduler.go` (renamed method call, updated log messages), `auction_ending_debug.go` (uses new query), `auction_lot_repository_test.go` (10 new test cases covering 23h, 12h, 2h, 25h, -1h windows, auction_end_time, mixed-case status, and Brian's exact scenario). All tests pass (`go vet` clean, `go test -v ./...` clean). **Lesson reinforced (THIRD TIME):** Auction `sale_date` is stored as UTC instant; users in negative-offset TZs lose lots whose end time crosses midnight UTC. Always use a rolling `now → now+24h` window instead of `[startOfDay, endOfDay)` for time-sensitive queries. Status is stored lowercase; UI displays uppercase — use `LOWER(status)` for robust comparison.
